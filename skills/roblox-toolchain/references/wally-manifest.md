# `wally.toml` Reference

Source: [github.com/UpliftGames/wally](https://github.com/UpliftGames/wally).

## Manifest fields

```toml
[package]
name = "scope/name"                          # lowercase letters, numbers, dashes; unique per registry
description = "Async primitives for Roblox"  # free-form; shown in listings + search
version = "2.0.7"                            # SemVer
license = "MIT OR Apache-2.0"                # SPDX expression — required for public publishing
authors = ["Name <email@example.com>"]       # free-form
realm = "shared"                             # "shared" or "server"
registry = "https://github.com/UpliftGames/wally-index"
homepage = "https://example.com/pkg"         # optional, shown on wally.run
repository = "https://github.com/x/y"        # optional source URL
exclude = ["node_modules"]                   # files to exclude from publish
include = []                                 # explicit include list (defaults respect .gitignore)
private = true                               # block publish entirely

[dependencies]
React = "jsdotlua/react@^17.1.0"
Promise = "evaera/roblox-lua-promise@4.0.0"

[server-dependencies]
ProfileService = "loleris/profileservice@1.4.0"

[dev-dependencies]
Jest = "jsdotlua/jest@3.10.0"
```

## Realms

Two values:

- `shared` — replicates to clients. Mount under `ReplicatedStorage`.
- `server` — server-only. Mount under `ServerStorage` or `ServerScriptService`.

There is **no `client` realm**. Client-only deps live in `shared` and are required by client code.

The realm of a package determines which output folder its installs land in:

| Section in `wally.toml` | Realm of dep | Installs into |
|---|---|---|
| `[dependencies]` | `shared` | `Packages/` |
| `[server-dependencies]` | `server` | `ServerPackages/` |
| `[dev-dependencies]` | either | `DevPackages/` (excluded from production builds) |

A `shared` package can be required by either side. A `server` package required from a client script will fail in Studio (the package isn't there).

## Version specifiers

Wally follows Cargo / npm SemVer:

- `"1.2.3"` or `"^1.2.3"` — `>=1.2.3, <2.0.0` (caret default).
- `"~1.2.3"` — `>=1.2.3, <1.3.0` (tilde, locked minor).
- `"=1.2.3"` — exact match.
- `"*"` — any version.

Verify exact semantics against [Wally docs](https://github.com/UpliftGames/wally) for non-default specifiers.

## Lockfile (`wally.lock`)

Generated by `wally install`. Pins exact versions of every dep, including transitives. Format:

```toml
[[package]]
name = "registry:scope/name"
version = "2.0.7"
dependencies = [
 "registry:other/dep",
 "git:https://github.com/owner/repo.git",
]

[[package]]
name = "registry:other/dep"
version = "1.5.0"
checksum = "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"

[[package]]
name = "git:https://github.com/owner/repo.git"
rev = "main"
commit = "e3b0c44..."
```

**Commit `wally.lock`.** CI runs `wally install --locked`, which fails if the lockfile is out of date with `wally.toml`. Catches "I forgot to commit the lock" before merge.

## Commands

| Command | Purpose |
|---|---|
| `wally init` | Scaffold a new `wally.toml`. |
| `wally install` | Resolve + download all deps; write lockfile. |
| `wally install --locked` | CI mode: error if lockfile is stale instead of regenerating. |
| `wally update [pkg...]` | Bump deps. Without args, updates everything. |
| `wally publish [--token <t>]` | Publish current package to the registry. |
| `wally login [--token <t>]` | Authenticate with a registry. Token can also come from `WALLY_AUTH_TOKEN`. |
| `wally logout` | Remove credentials. |
| `wally package [--list] --output <path>` | Build the publish zip for inspection / debugging. |
| `wally manifest-to-json` | Print the manifest as one JSON line (used by registry tooling). |
| `wally search <query>` | Search the registry. |

## Registries

A registry has two pieces:

- A Git repo holding a package index (the canonical wally registry: `github.com/upliftgames/wally-index`).
- A registry API serving package downloads + metadata.

API endpoints (for self-hosted registries):

- `GET /v1/package-contents/<scope>/<name>/<version>` — package zip
- `GET /v1/package-metadata/<scope>/<name>` — metadata
- `GET /v1/package-search?query=...` — search
- `POST /api/v1/publish` — upload a package tarball

You can run a private registry with the same protocol — useful for closed-source code shared across multiple games.

## Mounting Wally output in Rojo

```json
{
  "tree": {
    "ReplicatedStorage": {
      "$className": "ReplicatedStorage",
      "Packages": { "$path": "Packages" }
    },
    "ServerScriptService": {
      "$className": "ServerScriptService",
      "Packages": { "$path": "ServerPackages" }
    },
    "ServerStorage": {
      "$className": "ServerStorage",
      "DevPackages": { "$path": "DevPackages" }
    }
  }
}
```

Don't mount `ServerPackages` under `ReplicatedStorage` — server packages aren't supposed to replicate.

## Common pitfalls

- Forgetting to commit `wally.lock`.
- Publishing from a dirty working tree — tag a commit, publish from CI.
- Setting `realm = "server"` on a package that should be shared — clients can't require it.
- Mounting only `Packages/` and trying to use server-realm deps — they need `ServerPackages/` mounted separately.
- Hand-editing files inside `Packages/` — overwritten on next `wally install`.
- Ignoring the `wally update` cadence — old transitive deps accumulate vulnerabilities.
